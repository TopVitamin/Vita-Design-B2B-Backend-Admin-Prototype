# 视觉与Token规则V1.1

这份文档不是品牌手册，也不是设计赏析。

它主要回答的是这些问题：

- 为什么这套后台看起来是高密度、克制、偏理性，而不是营销页
- 为什么颜色、留白、圆角、阴影都要收着用
- 为什么不能在不同页面里各自发明一套按钮、Tag、Tab和间距
- 真正参与构建和部署的那两份基线代码到底在哪里

如果你只想快速判断“页面看起来为什么是这样”，先记住这3个部分是一起工作的：

- 本文档：负责讲视觉方向和使用边界
- [prototype-app/src/ui-foundation.css](../src/ui-foundation.css)：负责讲真实变量和值
- [prototype-app/tailwind-preset.ts](../tailwind-preset.ts)：负责讲这些变量在Tailwind里怎么用

如果你当前想判断的是“不同切换器该长成哪种风格”，优先看[切换器与Tabs样式规则 V1.1.md](切换器与Tabs样式规则%20V1.1.md)。

换句话说：

- 想改规则，先看本文档
- 想改具体样式值，去改`prototype-app/src/ui-foundation.css`
- 想改Tailwind里的调用方式，去改`prototype-app/tailwind-preset.ts`

如果你时间不多，先记住这4句话：

- 视觉服务于结构和效率，不服务于“做得花”
- 同一套后台里，不允许每个页面各搞一套间距、圆角和颜色
- 语义色只负责表达状态，不负责承担品牌表达
- 主题切换如果存在，默认只覆盖强调色和交互反馈层，不把整页中性色背景一起染色
- 要部署、要上GitHub、要过Vercel，就只认`prototype-app`里的基线代码

## 1. 视觉方向

先把风格定位说白一点：

风格定位：现代克制型高密度后台。

目标：

- 稳定
- 清晰
- 有秩序
- 信息密度高但不拥挤

禁止：

- C端卡片流
- 营销感文案
- 过强装饰
- 大面积品牌色铺底
- 为了“设计感”牺牲结构和效率

## 2. 基础视觉原则

这部分可以理解成“无论做哪个业务页面，视觉上都不要跑出这个边界”。

- 用边框、分组、留白和层级建立秩序
- 用少量语义色表达状态
- 页面背景、容器背景、悬浮层背景最多3层
- 标题、模块标题、正文、辅助文本最多4层文字层级
- 主题存在多套预设时，应拉开风格差异，避免同一轮里堆两三套极其接近的蓝色

## 3. 布局与密度

这里不是在追求“越紧越高级”，而是在追求“信息多，但读起来不乱”。

- 页面左右留白默认24px，上下留白默认24px
- 区块内边距默认16px
- 列表页筛选区宽屏默认6列
- 新增/编辑/详情卡片内字段宽屏默认4列
- 表格行高默认36px~40px
- 按钮和输入控件默认高度32px
- 圆角默认只在4px~12px之间递进，主卡片、筛选区、表格容器默认8px圆角
- 阴影默认只保留xs、sm、md三档，普通卡片优先xs或sm，浮层再升到md
- 顶部工作台页签条保持高密度，小字号、48px高度，不做浏览器式大页签
- 顶部壳层左侧品牌头与右侧页签条共用同一高度和底边线，默认白底对齐
- 顶部工作台Tab中的文字、icon和关闭按钮必须垂直居中；关闭按钮默认保留稳定点击热区，不依赖额外padding“看起来差不多”
- 顶部工作台Tab的关闭按钮默认只在hover或focus时显示，不常亮占据视觉注意力
- 顶部工作台Tab在宽度不足时默认收纳到“更多页签”下拉，不使用横向滚动条作为主要溢出方案
- 工作台Tab激活下划线默认只跟随文字区域，不延伸到关闭按钮区域
- 页面滚动默认发生在header以下的内容区，壳层header固定在顶部；避免滚动条贴着整页最右侧贯穿整个视口
- 顶部页签文字默认14px常规字重，常规操作按钮默认常规字重
- 顶部用户入口默认使用单独头像按钮承载，优先32px方形圆角头像；不额外包一层白底胶囊，也不默认附带下拉箭头
- `label`与控件的上下间距保持紧凑，默认4px~6px，V1.1基线取6px
- 查询区字段栅格默认横向16px、纵向12px左右的节奏，不为了“呼吸感”把筛选区拉成松散表单
- 壳层内消息预览头部、任务中心工具条这类系统级切换行优先保持紧凑高度，默认约48px；其中分段控件本身优先与32px输入框/按钮同高
- 列表工具条默认采用紧凑密度和明确分组，左侧批量操作、右侧弱操作之间通过弱间距区分，而不是依赖大块留白
- 左侧导航收起态默认只保留线性图标，不保留文字缩写块；展开/收起控制放在导航底部，不伪装成业务菜单项
- 左侧导航展开态一级分组之间默认保持紧凑节奏，不做过大的纵向空白；分组与分组之间优先通过弱间距和细分割线区分，而不是靠大块留白
- 左侧三级导航的视觉层级默认应为：一级分组 > 二级目录 > 三级页面；不允许二级目录因为字号过小而比三级页面更弱

间距使用约定：

这几条尤其重要，因为很多页面一乱，往往就是从spacing乱改开始的。

- `--space-*` 是原子Token，不等于应该直接覆盖Tailwind数字spacing刻度
- Tailwind里的`gap-2`、`p-4`、`px-3`应保留原生语义，不允许通过Token改造成2px、4px、12px这类异常缩小值
- 设计Token在Tailwind里通过语义别名暴露，例如`gap-control`、`gap-actions`、`gap-field`
- 行高别名统一使用`leading-ui-tight`、`leading-ui-normal`、`leading-ui-relaxed`，避免暴露看起来像原生但实际不可预测的命名

## 4. 基线资产与代码位置

如果你是来做部署、上传GitHub或者排查Vercel构建问题，这一节最重要。

V1.1正式使用以下2个技术基线文件，它们也是**唯一可执行真相源**：

- [prototype-app/src/ui-foundation.css](../src/ui-foundation.css)
- [prototype-app/tailwind-preset.ts](../tailwind-preset.ts)

统一口径：

- `ui-foundation.css`同时承载设计变量与语义布局类，不再拆成两份CSS文件
- `tailwind-preset.ts`是唯一Tailwind映射真相源，不再并存第二份手工副本
- `01-页面设计基线`目录现在只保留规则文档，不再保留第二份可执行CSS/TS代码

一句话记忆就是：

- 想理解为什么这样设计，看本文档
- 想让代码真正生效，改`prototype-app`里的那两份文件

## 5. 组件使用原则

这一段不是让你背组件清单，而是为了防止页面越做越像“拼装出来的杂货铺”。

- 状态统一用Tag / Badge
- 列表和明细统一用表格承载
- 详情信息用卡片化描述区
- 列表主数据区默认是“卡片容器+表格”的组合，不拆成营销式卡片流
- 数据展示默认遵循统一格式口径：金额优先2位小数，完整时间优先`YYYY-MM-DD HH:mm:ss`
- 抽屉用于轻量查看/补录
- 弹窗用于确认、轻表单、导入导出流程
- 主题切换弹窗属于轻配置弹窗，默认控制宽度和高度，不为了“展示案例”堆大面积预览区；优先展示主题名、风格说明、关键色板和应用动作
- 全局搜索结果面板默认跟触发输入框同宽对齐，不额外拉成超宽下拉层，避免压迫右侧通知和用户入口
- 分段控件允许采用胶囊激活项，但外层不应再套明显灰底边框壳；如果页面本身已有容器边界，分段控件优先直接悬浮在容器内
- 查询区和表单区必须按字段语义混合使用`input`、`select`、日期控件、`switch`、`checkbox`，不能整页几乎全是`input`
- 占位文案默认统一：`input`/`textarea`使用`请输入`，`select`/日期/时间控件使用`请选择`；该规则同样适用于表格内嵌编辑态
- `checkbox`文案过长时默认保持单行，超长截断并按需提供悬停完整文案，不让勾选项自己换行撑高整行布局
- 原生`checkbox`的勾选色默认跟随当前主题主色，不允许出现页面已切主题但勾选框仍保持旧蓝色或浏览器默认色的情况
- `checkbox`和`radio`在不同主题下默认应使用可控样式，不依赖浏览器原生主题算法；亮色主题中勾选符号或中心点默认保持白色，避免出现发黑或脏色
- 表单基础控件优先复用共享Input、Textarea、Checkbox、RadioGroup、Switch，不再长期散落原生标签和页面级class组合
- 详情页与只读展示优先使用文本展示，不用disabled表单控件冒充详情态
- 详情页Tab默认采用线性下划线式，不用胶囊按钮式Tabs
- 表单控件必须显式定义focus态，不能只给默认边框而没有交互反馈
- 图标风格统一使用简洁线性图标；最小运行工程基线采用`lucide-react`风格
- 后台原型中的`select`默认不直接依赖浏览器原生下拉面板；应统一使用可控的自定义下拉触发器与选项面板，避免不同浏览器出现箭头贴边、展开菜单缩进漂移、样式不一致等问题
- `select`触发器和选项文案默认保持单行，超长截断并按需用悬停补完整内容；不允许多行换行把下拉项高度撑乱
- `select`在接近视口底部时应自动翻转向上展开，并脱离父容器裁切；打开时不应引起页面抖动或布局跳动
- 若因实现成本暂时保留原生`select`，也必须隐藏浏览器原生箭头，统一使用线性下拉icon，并为右侧icon预留稳定内边距，不让icon贴边
- 弹窗和抽屉右上角关闭默认使用线性icon按钮，不使用“关闭”文字按钮占位
- 顶部用户菜单中的占位能力，例如个人中心、语言切换、退出登录，不使用浏览器原生`alert`；默认复用系统内顶部居中轻提示，并按错误或警示语气呈现
- 弹窗内的字段勾选项、格式选择项等辅助选项默认使用小字号正文，不与分区标题使用同一级视觉权重
- 导入弹窗选择阶段默认使用“说明Banner + 模板卡 + 上传区 + 模式切换 + 底部按钮”的紧凑节奏，不把每一块都拉成厚重容器
- 导入结果区的统计卡默认数字层级高于标签层级，并使用等宽数字对齐，不让三张卡的视觉重心飘散
- Tag / Badge默认采用轻圆角矩形，不使用过于圆润的全胶囊形态；后台状态标签优先稳定、克制，而不是“可爱感”
- 消息中心详情默认采用覆盖式右侧抽屉，依靠遮罩与阴影建立层级，不通过改变主列表宽度制造“伪抽屉”效果
- 分页默认采用精简形态，优先展示总数、每页条数、上一页/下一页、页码信息和跳转，不默认堆满复杂页码按钮
- `列设置/表格设置`默认使用抽屉或大尺寸弹窗承载，不做悬浮小气泡；当字段分组、排序和固定列同时存在时，优先用左右分栏结构
- `列设置`面板内默认支持分组勾选、已选列表排序与`恢复默认`；字段多时左侧按业务分组展示，右侧展示“已选列”；字段特别多时再评估是否需要字段搜索
- `列设置`中的已选列排序应可拖拽调整；拖拽手柄、勾选框、固定标记等控制元素保持线性、弱装饰风格
- `列设置`默认不强调“保存为模板”；主按钮文案应为`保存并应用`或`确定`，表达“修改当前用户配置并立即生效”
- 若支持表格行高切换，默认使用`紧凑 / 中等 / 宽松`三档分段控件，不与列勾选区域混排
- `列设置`里的辅助说明默认放在右侧已选区或底部说明位，不在主内容区堆大段帮助文本
- 表头里的排序icon默认应与文字垂直居中，尺寸保持紧凑，不因过大或过远拉坏表头节奏
- 右对齐表头在排序icon和右侧列边界之间默认预留一层缓冲，不允许icon贴边
- 表头列分割线默认使用短竖线，居中且高度低于整行高度，避免做成通栏硬切
- 表头分割线、拖拽热区和悬停高亮默认使用同一条边界线，不允许默认灰线与hover高亮并排出现双线
- 表体默认不强调竖向列分割线；宽表格优先通过表头分割、列宽和横向留白建立列边界感
- 左侧导航滚动区域默认允许滚动但可隐藏滚动条本体，避免常驻滚动条破坏壳层整洁度
- 侧栏顶部租户切换卡位默认保持紧凑节奏：图标、名称、编码三者间距收紧；公司名称默认单行截断，不因换行把顶部区域撑厚
- 样例中的调试/演示工具使用弱化悬浮面板承载，默认低透明度、低对比度，不与真实业务按钮共享同一视觉层级

## 6. 与最小运行工程的关系

很多人看到这里会有一个误区：以为“规则文档在这里，代码真相源也应该在这里”。现在不是这样了。

根目录的`prototype-app`就是V1.1的唯一运行工程，也承载了唯一的CSS基线与Tailwind preset代码文件。

这个目录负责解释规则，`prototype-app`负责真正运行和部署。两者关系应理解为：

- 文档在这里讲清楚边界和原则
- 代码在`prototype-app`里真正落地
- 如果你改了基线实现，必须同时回看本文档，避免说明和代码长期漂移

V1.1最小运行工程同时提供一个`Design System`单页总览，作为视觉基线的可视化展示入口，用来说明这套后台最终稳定可复用的视觉与控件约束长什么样。

如果最小运行工程提供主题切换，则`Design System`总览默认应能说明主题覆盖边界，而不是只展示单一蓝色主题。推荐至少覆盖这些维度：

- 主品牌交互色：`--primary`、`--primary-hover`、`--primary-active`、`--primary-subtle`
- 页面层级色：默认说明`--bg-page`、`--bg-subtle`、`--bg-hover`保持统一中性色，仅`--bg-active`可随主题变化
- 边框层级：默认说明`--border-default`、`--border-subtle`、`--border-strong`保持统一中性色，仅`--border-focus`随主题变化
- 链接与处理中状态：`--link*`、`--tag-processing-*`

该总览页默认应覆盖以下内容：

- 颜色
- 字体
- 间距与密度
- 圆角与阴影
- 卡片容器
- 表格基线
- 数据格式示例
- 壳层与反馈
