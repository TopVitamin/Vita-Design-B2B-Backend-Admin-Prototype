# 页面设计基线

> **文档副本：** 本目录是仓库根目录 `01-页面设计基线` 的备份副本，便于单独将本工程推送到 GitHub 时仍能查阅规则。你在本地仍以根目录下同名文件夹为与 `prototype-app` 平行的权威说明位；改完规则后若需对外一致，请将更新同步复制到此处。

如果你关心的是下面这些问题，就应该先来这个目录：

- 页面结构为什么这么摆
- 信息为什么这样分区
- 控件为什么是这种密度和样式
- 我想改样式和Token时，应该先看哪些规则

这里仍然是整套仓库里**最直接影响页面展示效果**的一层，但它现在只负责**解释规则和边界**，不再保存第二份可执行CSS/TS代码。

## 先记住这件事

真正参与构建、上传GitHub和部署Vercel的基线文件，只有这两份：

- [prototype-app/src/ui-foundation.css](../src/ui-foundation.css)
- [prototype-app/tailwind-preset.ts](../tailwind-preset.ts)

这个目录只负责告诉你：

- 为什么这些值要这样定
- 哪些地方能改，哪些地方不要乱改
- 页面结构和视觉风格的边界是什么

## 最近已经沉淀下来的通用规则

这几条是最近从真实案例里反复验证后，已经可以当通用规则使用的：

- 查询区默认按桌面端6列栅格计算，最多展示2行、总占位不超过12列；跨列条件按实际占位列数计数，超出部分才进入展开区
- 纯查询报表页不额外补无意义的`刷新`和装饰性的“共X条记录”
- 宽表格必须按字段语义分配列宽，并评估固定列、截断和等宽数字
- 样例调试面板必须和业务操作区分离，且默认支持鼠标拖动
- 全局搜索默认只搜左侧菜单，并支持`/`、`↑ / ↓`、`Enter`、`Esc`这套键盘交互
- 消息详情这类查看态优先使用覆盖式抽屉，不用分栏硬挤主体内容
- 切换器不能全站只用一种样式，必须区分工作台Tab、状态Tab、内容Tab、分段控件和筛选Chip
- Select触发器和选项统一单行截断，靠近底边时自动翻转，不能因为打开下拉导致页面抖动
- 导入弹窗的选择阶段、加载阶段和结果阶段优先复用共享骨架，不再每个业务页各写一套模板卡和上传区
- 列表页优先复用PageHeader、ListPageMainCard、ListPageToolbar和Pagination这套外层骨架
- 顶部壳层默认固定在顶部，滚动发生在header以下内容区
- 查询区如果只承担筛选和操作，不默认再写“查询筛选区”标题

## 这个目录建议再补强的重点

现在这层已经比之前清楚很多了，但后面如果继续完善，我建议优先盯这3件事：

- 把“哪些属于跨项目稳定规则，哪些只是当前案例实现”写得更明确
- 把“列表页、详情页、编辑页”每种页面的ASCII骨架继续补全
- 把“哪些是假交互绝对不允许出现，哪些未开放功能必须给轻量反馈”继续收紧成固定口径

## 这个目录不负责什么

- 不负责某个具体业务页面的字段和状态
- 不负责Prompt怎么写
- 不负责验收清单怎么组织
- 不负责承载第二份可执行CSS/TS代码

这些内容在完整仓库中还包含「02-怎么使用这套项目」目录；仅上传本工程时请以根目录 [README.md](../README.md) 为准。
