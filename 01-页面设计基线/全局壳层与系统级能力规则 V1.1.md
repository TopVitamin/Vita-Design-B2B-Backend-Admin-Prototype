# 全局壳层与系统级能力规则V1.1

这份文档不负责讲具体业务字段。

它主要回答的是这些问题：

- 顶部壳层里哪些能力属于系统级，而不是某个业务页自己的特例
- 为什么全局搜索、消息通知、用户菜单、工作台Tab要统一放在壳层处理
- 查看态详情为什么优先用覆盖式抽屉，而不是分栏硬挤主体
- 哪些系统级交互应该成为默认规则，而不是只存在某个页面代码里

如果你要看具体实现位置，优先对照以下文件：

- [prototype-app/src/components/app-shell.tsx](../src/components/app-shell.tsx)
- [prototype-app/src/pages/message-center.tsx](../src/pages/message-center.tsx)
- [prototype-app/src/components/ui/floating-alert.tsx](../src/components/ui/floating-alert.tsx)

## 1. 适用范围

本规则默认适用于以下系统级能力：

- 全局搜索
- 消息通知预览
- 导出任务中心入口
- 用户菜单
- 工作台Tab
- 顶部轻提示
- 覆盖式查看抽屉
- 消息中心这类系统级列表页

这些能力如果已经进入后台壳层，就不应再被当作单页局部玩法处理。

## 2. 顶部壳层组成

标准壳层默认包含：

- 左侧品牌头
- 工作台Tab区
- 全局搜索
- 导出任务中心
- 消息通知
- 用户菜单

基本规则：

- 左侧品牌头与右侧Tab区同高、同底边线
- 顶部壳层默认固定在视口顶部，页面内容在header以下独立滚动；不让整页`body`滚动穿过header
- header和内容区之间默认保留轻阴影或弱边界，用来稳定固定头部的层级感
- 首页Tab默认存在且不可关闭
- 系统级能力入口优先放在顶部壳层，不混入业务标题区
- 用户入口和通知入口保持高密度、小尺寸、弱装饰
- 下载 / 通知 / 用户入口彼此独立排列，不允许因为嵌套容器造成视觉黏连
- 顶部搜索框、导出、通知、用户头像这排控件默认按整条header高度垂直居中，不通过额外上下padding做“假居中”

## 3. 全局搜索

全局搜索默认是“菜单搜索”，不是“整页数据搜索”。

固定规则：

- 只搜索左侧可用菜单
- 不搜索顶部页签、帮助、用户菜单或其他全局动作
- 支持点击输入框触发
- 支持`/`快捷键唤起
- 当结果不止一项时，支持`↑ / ↓`切换高亮、`Enter`跳转、`Esc`关闭
- 搜索结果面板默认与输入框同宽，不额外拉成超宽浮层
- 搜索结果项优先展示两行：菜单名称、菜单路径
- 搜索结果点击后直接跳到对应菜单，不新增二次确认

不建议：

- 把全局搜索做成“什么都搜”的混合搜索
- 在结果里混入帮助、主题切换、退出登录这类非菜单动作
- 结果面板做得比输入框宽很多，压迫通知和头像区域

## 4. 消息通知

顶部消息通知默认采用“铃铛入口 + 未读徽标 + 预览面板”。

固定规则：

- 铃铛入口显示总未读数
- 预览面板默认支持`消息 / 通知 / 待办`三类切换
- 预览面板底部操作默认保留`全部已读`和`查看更多`
- 不在预览面板顶部重复再放一颗“全部已读”
- 预览面板里的单条消息点击后，进入消息中心并打开对应详情
- 预览面板只承载“快速预览和快速进入”，不在这里堆复杂处理流程
- 预览面板头部切换区保持紧凑，不因上下padding过大把整行撑高
- 面板中的`消息 / 通知 / 待办`允许使用胶囊分段，但不额外包一层厚重灰底外壳

文案约束：

- 使用`待办`，不使用`代办`
- 未开放或占位动作仍需给系统级反馈

## 5. 用户菜单

用户菜单默认采用单独头像入口。

固定规则：

- 头像入口优先使用32px方形圆角头像
- 可使用系统缩写，如`VM`
- 不额外包白底胶囊
- 不默认显示下拉箭头
- 个人中心、主题切换、语言切换、退出登录为默认顺序
- 非主题切换类占位动作默认复用顶部轻提示，不用浏览器原生`alert`

## 5.1 SaaS租户切换

SaaS系统里的租户切换默认属于全局上下文切换，不应被理解成用户偏好设置。

固定规则：

- 当前租户应持续可见，不能只在下拉层里短暂出现
- 如果系统采用强左侧导航壳层，租户切换入口默认放在左侧导航顶部、菜单区之上
- 如果系统没有强左侧导航而以顶部Header为主，租户切换入口默认放在顶部左侧Logo或产品名右侧
- 租户切换入口默认不放在左下角；左下角更适合帮助、设置、版本、退出等低频能力
- 租户切换默认支持搜索租户名称或编码；租户很多时不应要求用户纯滚动查找
- 租户切换后应联动刷新数据范围、权限、菜单和页面上下文；若页面存在未保存内容，应先做切换确认
- 当前原型工程采用“左侧导航顶部租户切换”方案，作为默认沉淀口径

## 6. 工作台Tab

工作台Tab属于系统级导航能力，不是页面装饰。

固定规则：

- 工作台Tab区默认48px高，左侧品牌头和右侧Tab区共用同一高度
- Tab中的文字、icon和关闭按钮必须垂直居中；关闭按钮应保留稳定点击热区，而不是只放一个裸icon
- 关闭按钮默认只在对应Tab hover或focus时显示，不常驻干扰阅读
- Tab激活下划线默认只跟随文字区域，不覆盖关闭按钮
- 首页Tab固定存在，不可关闭
- 业务Tab默认可关闭
- 同一业务页默认单实例打开
- 消息中心、壳层能力参考页这类系统页也可作为独立Tab存在
- 关闭当前Tab后，焦点回到前一个仍存在的Tab
- 当工作台Tab宽度不足时，默认把超出项收纳到“更多页签”下拉，并显示折叠数量；不依赖横向滚动条承接溢出

## 7. 顶部轻提示

顶部轻提示是系统级轻反馈统一口径。

固定规则：

- 默认位于页面顶部居中
- 默认约2秒自动消失
- 不打断当前流程
- 成功 / 提醒 / 错误三类语气区分明确
- 壳层级占位能力和系统反馈统一复用该组件

不建议：

- 退回浏览器原生`alert`
- 用大弹窗承载“未开放”“已切换”这类轻提示

## 8. 覆盖式抽屉

查看态详情优先使用覆盖式抽屉。

固定规则：

- 抽屉默认从右侧覆盖在主体内容之上
- 带遮罩和阴影，明确层级
- 遮罩覆盖整个视口，而不是只覆盖页面卡片区域
- 主列表或主页面宽度保持稳定，不被硬挤压
- 右上角关闭默认使用线性icon按钮
- 查看态详情、消息详情、轻量核对类页面优先采用抽屉

不建议：

- 用分栏把主列表挤窄后，假装成抽屉
- 为了查看一条消息或一条记录，跳整页详情

## 9. 消息中心骨架

消息中心默认采用以下结构：

```text
[页面标题区]

[左侧分类导航] [中间消息列表] [右侧详情抽屉（覆盖式）]
```

固定规则：

- 左侧分类导航承载消息分类和未读数
- 中间列表承载筛选、批量已读和消息列表
- 右侧详情使用覆盖式抽屉，不改变列表宽度
- 详情抽屉头部和中间列表工具条的分割线要对齐
- 详情抽屉内如有“查看采购单”“查看库存流水”等动作，作为次级动作放在详情内

## 10. 导出任务中心

导出任务中心默认属于系统级任务页。

固定规则：

- 顶部壳层提供独立下载icon入口
- 状态切换更接近“模式/过滤切换”，默认使用分段控件，不强制走线性Tab
- 状态切换所在工具条保持紧凑，整行优先控制在48px左右
- 导出任务详情与消息详情一样，默认使用覆盖式抽屉，不在卡片内局部展开
- 下载、重试、打开来源页等动作优先在任务详情抽屉内完成

## 11. 沉淀方式

以后新增系统级能力时，优先按以下顺序沉淀：

1. 先写入本文档
2. 再写入`页面规格说明模板.md`和`AI输入输出说明.md`
3. 再补进`验收清单.md`
4. 最后决定是否进入`03-完整案例`和`prototype-app`

如果某个系统级能力已经在代码里存在，但还没有进入案例和模板，那它还不算真正完成沉淀。
