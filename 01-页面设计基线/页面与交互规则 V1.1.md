# 页面与交互规则V1.1

这份文档不是拿来讲颜色和Token的。

它主要回答的是这些很实际的问题：

- 为什么这个页面不能做成C端卡片流
- 为什么按钮要放在这里，而不是随便堆一排
- 为什么详情页要这样分区
- 为什么状态、异常、权限不能只在接口里存在，必须在页面上被看见

它决定的是：

- 页面骨架
- 信息分区
- 操作层级
- 状态、异常、权限在页面上怎么被看见

它不负责颜色和Token值，那部分在[视觉与Token规则 V1.1.md](视觉与Token规则%20V1.1.md)。
如果你要判断“这里到底该用线性Tab、分段控件还是筛选Chip”，优先看[切换器与Tabs样式规则 V1.1.md](切换器与Tabs样式规则%20V1.1.md)。

如果你时间不多，先记住这4句话：

- 后台页面先解决效率和秩序，再谈“好不好看”
- 页面结构要稳定，不能每个页面都重新发明一套摆法
- 用户看见的按钮、开关、Tab、全选，不能做成没反应的假交互
- 状态、异常、权限、上下游和记录，不是补充项，而是默认必做项

## 1. 总体原则

先把大方向卡住，后面具体页面再往下细化：

- 默认是PC端高密度后台，不是C端产品页
- 优先表格、表单、标签、抽屉、弹窗、分页
- 不用大留白、大卡片流、营销化文案、花哨流程图
- 所有页面都必须显式设计状态、异常、权限、上下游与记录

### 1.1 哪些能上升成稳定规则，哪些还只是当前案例实现

这里很容易混淆。

不是某个案例做得顺手，就立刻可以写进通用规则。更稳的判断方式是：

- **能上升成稳定规则的**：跨项目、跨页面类型、跨业务对象仍然成立，写进规范后不会把后续页面做死
- **暂时只是当前案例实现的**：只对某个业务对象、某个案例链路、某一组状态或某一种字段结构成立

一条案例经验想正式上升成稳定规则，至少要同时满足这3个条件：

- 不依赖某个特定业务名词，换成别的业务也成立
- 人工接手和AI执行时都能稳定理解，不容易误读
- 写进规范后，会提高一致性和交付效率，而不是制造更多例外

可以优先上升成稳定规则的，通常包括这些：

- 页面骨架顺序
- 操作层级
- 查询区与工具条的摆放关系
- 状态Tab的典型位置和行为
- 轻量反馈、无权限、空状态、加载中这些共通交互
- 列设置、分页、详情Tab这类高频后台能力

通常还只能算案例实现的，往往是这些：

- 采购订单、客户、供应商各自的状态名称
- 某个页面默认选中哪个Tab
- 某个页面有哪些批量操作
- 某个业务对象的字段数量、字段顺序、字段分组
- 某个案例里为了演示而加的临时说明、示例数据、测试入口

如果一条案例经验已经在多个页面里反复成立，也可以上升成稳定规则。推荐做法是：

1. 先在案例里验证它确实好用
2. 再判断它是不是跨项目仍然成立
3. 最后再写回本文档，而不是直接把案例代码当规则

## 2. 全局壳层

这一段讲的是整套系统共用的外壳，不是某一个业务页面自己的特殊玩法。

- 左侧导航：最多3级，默认展开，可折叠；展开/收起控制放在导航底部
- 左侧导航收起后保留一级菜单图标，展开后支持一级、二级分组逐级展开/收起
- 展开态左侧导航默认保持清晰的三级阅读层级：一级分组最强、二级目录次之、三级页面作为正文层级，不允许二级目录反而比三级更小更弱
- 顶部区域：页签栏 + 全局搜索 + 用户辅助区；左侧品牌头与右侧页签条同高、同底边线
- 顶部壳层默认固定在视口顶部，页面滚动应发生在header以下的内容区，而不是整页`body`
- 壳层header和内容区之间默认保留清晰边界，可使用轻阴影或弱分界线建立层级，但不做厚重悬浮条
- 用户辅助区默认优先放“单独头像入口”，不是一整块白底胶囊工具组；头像按钮默认直接可点并展开菜单
- 顶部头像入口默认使用32px方形圆角头像，可使用系统缩写如`VM`；不额外显示下拉箭头，避免重复暗示
- 顶部用户菜单如包含个人中心、主题色切换、语言切换、退出登录，默认顺序即为这4项；主题色切换进入弹窗，其余占位能力走轻量反馈
- 如果包含全局消息通知，默认采用铃铛入口 + 未读徽标；点击后打开顶部预览面板，而不是直接跳整页
- 消息预览面板默认支持`消息 / 通知 / 待办`三类切换，预览面板底部操作默认保留`全部已读`和`查看更多`
- 消息预览面板中的单条消息点击后，默认进入消息中心并打开对应详情，不在预览层里承载复杂处理流程
- 顶部页签默认第一个为首页页签，带Home类图标且不可关闭，用作系统着陆页和回退锚点
- 页签行为：同路由单实例；业务页签默认可关闭；有未保存内容时关闭或刷新需提示
- 列表页签属于业务页签，不因“常用”而变成不可关闭；关闭后应可通过首页或左侧导航重新打开
- 新建页、编辑页、详情页应视为独立任务页签，可并行存在，不把新建和编辑压成同一页签
- 全局搜索：默认只搜菜单，不替代列表筛选
- 全局搜索默认不检索顶部页签、帮助、用户菜单或其他全局动作，只检索左侧可用菜单
- 全局搜索默认支持点击触发和`/`快捷键唤起；当结果不止一项时，支持`↑ / ↓`切换高亮、`Enter`选中跳转
- 全局搜索结果面板默认与输入框同宽对齐，不额外拉成超宽浮层；结果项以“菜单名 + 菜单路径”两行信息为主，不重复堆栏目名
- 主题切换如果出现在全局壳层，应视为整套后台的全局偏好配置，而不是某个单页内的局部换肤

## 3. 页面类型骨架

### 3.1 列表页

把列表页理解成一种很稳定的阅读顺序：

先看页面范围和操作，再看筛选条件，再看表格，再看分页。

ASCII示意图：

```text
[顶部页签栏]

[页面标题区 + 页面级主操作]
  页面标题 / 当前对象范围 / 页面级主动作 / 页面级次动作

[查询筛选区]
  常驻筛选项
  展开后附加筛选项
  右侧操作：展开/收起 -> 重置 -> 查询

[批量工具条 / 表格工具条]
  左侧：已选条数 / 批量动作
  右侧：导出 / 列设置 / 密度切换（如有）

[表格数据区]
  表头
  数据行
  行内操作

[分页区]
  总数 / 每页条数 / 当前页信息 / 上一页下一页 / 跳转
```

固定结构：

1. 顶部页签栏
2. 页面标题区
3. 查询筛选区
4. 表格上方批量工具条
5. 表格数据区
6. 分页区

规则：

- 页面级操作在标题区
- 列表页外层骨架默认优先复用`PageHeader`、`ListPageMainCard`、`ListPageToolbar`和`Pagination`，不从零拼装每个页面的外层容器
- 查询区默认按桌面端6列栅格计算，最多展示2行、总占位不超过12列；单列条件按1列计，时间范围等跨列条件按实际占位列数计；只有超出12列占位时，才通过“展开/收起”承载附加筛选项
- 查询区如果只承担筛选控件和右侧操作，不额外承载说明文案时，默认可以省略“查询筛选区”标题，让卡片本身承担分组作用
- 查询区字段必须按语义使用混合控件：枚举/状态优先`select`，日期优先日期控件，布尔条件优先`switch`或`checkbox`
- 查询区、表单区、表格内嵌编辑区的控件默认占位文案必须统一：`input`/`textarea`用`请输入`，`select`/日期/时间控件用`请选择`
- `checkbox`文案较长时默认单行显示，超长截断，不通过自动换行把整行控件高度撑开；需要完整文案时优先用悬停提示补足
- 查询区“展开/收起”默认和“重置/查询”放在同一右侧操作组，不单独漂在左侧
- 查询区右侧操作组默认顺序为“展开/收起”→“重置”→“查询”
- 查询区默认保持高密度节奏：字段行距优先控制在约12px，控件组与操作组之间优先使用一层弱间距，不拉成松散的双层留白
- 批量操作只放工具条，不混进查询区
- 纯查询报表型列表页如果没有勾选和批量操作，可以不单独保留左侧批量工具区，只保留右侧表格工具条
- 行内操作优先2~3个，高风险动作进“更多”
- 页面上已经呈现为可点击/可切换的控件，不允许做成无反馈的假交互；要么具备最小可用行为，要么明确禁用/置灰并说明原因
- 暂未规划但暂时保留入口的动作，例如打印、历史成交，不应点击后静默无响应；默认给出轻量反馈
- 这类轻量反馈默认采用页面顶部居中的浮层Alert，停留约2秒后自动消失，不打断当前页面流程；提示语气按成功/提醒/错误区分
- 系统内已有统一轻提示时，占位反馈默认复用该提示体系，不再退回浏览器原生`alert`
- 用户菜单中的占位动作，例如个人中心、语言切换、退出登录，默认使用错误或警示语气的顶部轻提示，明确告知“暂未实现/暂未开放”
- 不应机械性给每个查询报表页都放一个`刷新`按钮；如果“查询”本身已经承担重新拉取最新结果的语义，页面头部不再重复放无意义的刷新
- 表格上方如果已经没有独立业务提示或批量信息，不要为了“看起来完整”再额外补一行`共X条记录`；总数信息默认放在分页区即可
- 列表页默认应评估是否需要`列设置`；字段较多、角色差异明显、不同岗位关注列不同的列表页，默认应提供`列设置`入口
- `列设置`入口默认放在表格工具条右侧，和刷新、导出等弱次级操作同组，不进入查询区，不抢主操作层级
- 列表工具条默认保持紧凑密度，整行优先控制在约48px~52px；左侧批量操作与右侧弱操作之间通过分组而不是大空白建立层级
- `列设置`默认支持：显示/隐藏列、列顺序调整、固定列设置；如页面确有需要，可附带表格行高切换，但不作为所有列表页默认必开项
- `列设置`默认不提供“保存为个人模板/公共模板”能力；基线方案应直接按`用户 + 页面 + 表格标识`自动记忆最近一次配置
- 用户记忆的配置至少应包含：显示列、列顺序、固定列、列宽、行高（如有）
- `恢复默认`必须显式提供，便于用户回到系统推荐列配置
- 固定列应遵循连续固定原则，默认只允许从左侧开始连续固定；固定列数量应设上限，避免冻结过多列破坏可读性
- 列表页重新进入、刷新、切页后，用户列配置应保持一致；切换账号或切换不同列表页时不共享配置
- 宽表格必须在页面规格中提前定义列宽策略，不允许所有列用同一默认宽度硬挤；日期、时间、单号、状态、动作、数量前后值都应按字段语义单独给宽度
- 宽表格默认应评估左侧固定列，优先固定识别性最强的列，例如时间、单号、编码、货主、仓库
- 宽表格里的长文本列允许截断并悬浮显示完整内容；数量列、时间列、单号列优先使用右对齐或等宽数字等稳定语法，避免横向阅读时节奏混乱
- 宽表格默认应支持列宽拖拽；拖拽后的列宽按`用户 + 页面 + 表格标识`自动记忆，不要求用户另存模板
- 表格排序默认只对数字、金额、日期、时间类字段开放，不对所有文本列机械性补齐排序icon
- 表格排序必须先作用于当前筛选结果的全量数据集，再进入分页；不允许只在当前页局部排序
- 表格排序默认支持升序和降序两态切换；若当前字段不适合排序，则不展示排序icon
- 表头分割线默认只在表头表达列边界，不要求在表体继续用竖线贯穿整列；表体优先保留横向行分隔，避免宽表格产生额外视觉噪音
- 表头分割线、列宽拖拽热区和悬停高亮应共用同一条列边界，不允许在同一位置叠出两根线
- 分页区至少包含总数、每页条数切换、上一页/下一页、当前页信息和跳转到指定页能力
- 分页默认保持精简，不把复杂页码按钮堆成主要视觉焦点

带Tab状态的列表页：

把它理解成“列表页的一个常见变体”就行，不要把它想成另一种完全不同的页面。

- 这仍然属于列表页，不是独立于列表页之外的新页面类型
- 适用于“状态分组是用户最高频切换维度”的场景，例如待提交、待审核、已完成
- 默认结构应理解为：

```text
[顶部页签栏]

[页面标题区 + 页面级主操作]

[查询筛选区]

[状态Tab区]

[批量工具条 / 表格工具条]

[表格]

[分页]
```

- 状态Tab区默认位于“页面级操作区或查询区之后、表格主体之前”，本质上是列表数据的状态分组导航
- 如果页面没有独立查询区，状态Tab区通常位于页面级操作区下方、表格工具条上方
- 状态Tab默认展示“状态名 + 数量”，例如`全部（134）`、`待审核（12）`
- 点击状态Tab的本质是切换当前列表的数据分组，不是打开新的业务页签
- 状态Tab默认单选，切换后应回到第一页；是否保留其他查询条件，必须在页面规格中明确
- 状态Tab默认采用线性下划线式，不使用胶囊按钮式；视觉层级低于页面标题和主操作，高于表格
- 状态Tab视觉默认保留一条满宽细分割线和当前激活项下划线，不额外叠出第三条重复边线
- 已经用状态Tab承载的高频状态分组，不应再在查询区重复堆一组完全相同的状态按钮

全局主题切换弹窗：

- 全局主题切换默认使用中尺寸轻配置弹窗，不把它做成接近整页的大弹窗
- 主题卡片优先展示主题名、简短风格说明、交互色板和必要的中性色说明；不强制放大面积页面示意图
- 应用主题后默认立即生效，并记住用户选择；同一次点击必须有明确反馈，不能切换后无任何提示
- 全局主题切换默认只切强调色，不默认改`bg-page / bg-subtle / bg-hover / border-default / border-subtle / border-strong`这类大面积中性色
- `checkbox`、focus边框、选中态和处理中标签等交互反馈层必须跟随主题联动，避免主题已切换但局部控件仍停留旧色

消息中心页：

- 消息中心默认采用“左侧分类导航 + 中间消息列表 + 右侧消息详情”的骨架
- 右侧消息详情默认使用覆盖式抽屉，不通过分栏硬挤中间列表区宽度
- 消息详情抽屉默认带遮罩、固定宽度和右上角关闭；关闭后列表布局应保持稳定，不发生重新排版跳动
- 消息详情中若存在关联业务入口，例如查看采购单、查看库存流水，默认作为抽屉内次级动作，不取代消息详情主体

### 3.2 新增/编辑页

新增页和编辑页的重点不是把字段堆满，而是让人知道“先填什么、再补什么、提交后会发生什么”。

ASCII示意图：

```text
[顶部页签栏]

[页面标题区 + 页面级主操作]
  页面标题 / 当前对象状态（如有） / 保存 / 提交 / 返回

[基础信息卡片]
  字段分组A
  字段分组B
  字段分组C

[表格明细卡片]
  区块标题
  区块级操作：新增行 / 批量导入 / 引用历史（如有）
  明细表格

[其他关键信息卡片]
  备注 / 附件 / 扩展字段 / 下游说明 / 风险提示

[底部操作栏（可选）]
  仅超长表单、向导式流程或顶部主操作脱离视口时出现
```

固定结构：

1. 顶部页签栏
2. 页面标题区
3. 基础信息卡片
4. 表格明细卡片
5. 其他关键信息卡片
6. 底部操作栏（可选）

规则：

- 标题区右侧放主操作
- 常规新建/编辑页默认只保留顶部主操作，不在顶部和底部重复放置相同保存/提交动作
- 底部操作栏仅用于超长表单、向导式流程或顶部主操作会脱离视口的场景，不作为默认形态
- 头表单统一`Label在上、控件在下`
- 表单字段必须按语义混合使用`input`、`select`、日期控件、`switch`、`checkbox`
- 表单控件默认占位文案统一使用基线规则：`input`/`textarea`为`请输入`，`select`/日期/时间控件为`请选择`
- 编辑页出现只读场景时，优先转入详情页语法或文本展示语法，不把整页简单处理成disabled表单
- 默认一行4列，窄屏递减
- 明细一律表格化编辑
- 明细区里显式出现的操作，例如新增行、批量导入、引用历史数据，不应做成空点击；要么具备最小可用行为，要么按轻量反馈规则处理

### 3.3 详情页

详情页不是“禁用版表单页”，而是“把关键事实清楚展示出来”的页面。

ASCII示意图：

```text
[顶部页签栏]

[单号标题区 + 页面级主操作]
  单号 / 状态 / 创建人 / 创建时间 / 组织 / 关键摘要
  页面级主动作 / 页面级次动作 / 更多

[卡片化信息区]
  基础信息卡
  业务信息卡
  结算或补充信息卡
  备注 / 说明 / 风险提示

[Tab区]
  明细 / 关联单据 / 日志 / 审批记录 / 其他扩展Tab

[当前Tab内容区]
  当前Tab对应的数据承载区
```

固定结构：

1. 顶部页签栏
2. 单号标题区
3. 卡片化信息区
4. Tab区

规则：

- 标题区左侧放单号、状态、创建人、创建时间、组织等关键元信息，右侧只放页面级操作
- 卡片内只读字段统一纵向文本展示，不用disabled输入框冒充详情态
- 明细、关联单据、日志、审批记录放Tab，不塞进标题区
- Tab默认采用线性下划线式切换，不用胶囊按钮式

### 3.4 导入弹窗

导入弹窗不是只负责“选个文件”，它必须把结果讲清楚。

ASCII示意图：

```text
[弹窗标题区]
  标题 / 关闭

[步骤1：选文件]
  模板下载说明
  上传控件
  导入范围或覆盖规则说明（如有）

[步骤2：导入中]
  处理中提示
  当前进度或状态说明

[步骤3：结果]
  成功摘要
  失败摘要
  行级失败明细下载或查看入口

[弹窗底部操作区]
  取消 / 关闭 / 重新导入 / 下载结果
```

固定结构：

1. 选文件
2. 导入中
3. 结果

补充规则：

- 导入弹窗三阶段默认优先复用共享骨架，不再让采购、客户、供应商各自维护一套模板卡、上传区和结果容器
- 选文件阶段默认包含：说明Banner、模板下载卡、上传拖拽区、结果演示或导入模式切换、底部操作区
- 上传拖拽区默认使用弱化虚线边框和轻辅助底色，不做厚重占位框
- 导入中阶段默认只保留一个明确处理态、简洁说明和居中布局，不重复堆列表和多组说明
- 结果阶段的摘要卡默认先强调数量，再承载标签；数字优先使用等宽数字，保证三张卡之间的阅读节奏稳定
- 行级失败明细如果存在，默认位于摘要卡之后、底部按钮之前，标题和下载动作放在同一行

结果至少覆盖：

- 全部成功
- 行级部分失败
- 文件级失败

### 3.5 导出弹窗

导出弹窗的重点是说清楚“导什么、按什么范围导、导成什么格式”。

ASCII示意图：

```text
[弹窗标题区]
  标题 / 关闭

[导出范围区]
  当前筛选结果 / 当前勾选结果 / 全部数据（按权限）

[导出字段区]
  默认字段说明
  可选字段勾选（如有）

[文件格式区]
  XLSX / CSV / 其他允许格式

[结果或提示区]
  导出任务创建成功提示
  大数据量异步导出说明（如有）

[弹窗底部操作区]
  取消 / 确认导出
```

固定结构：

1. 导出范围
2. 导出字段
3. 文件格式
4. 确认导出

## 4. 操作层级

这一段是为了避免一个页面上所有按钮都在抢视觉，最后谁都不重要。

- 页面级主操作：1个最突出
- 页面级次操作：1~3个
- 批量操作：依赖勾选，不常亮抢视觉
- 行内操作：查看 > 编辑 > 执行 > 危险/关闭
- 查询区只放查询、重置、展开，不放业务主动作
- 顶部页签文字和常规操作按钮默认使用常规字重，不通过加粗制造层级
- `列设置`属于表格级次操作，不视为页面级主动作；默认层级低于新增、导入、导出等业务动作

## 5. 状态、异常、权限

很多页面之所以看起来像“概念稿”，不是因为视觉不够好，而是因为这三类东西没补齐。

每个页面默认必须补齐：

- 加载中
- 空数据
- 查询无结果
- 无权限
- 接口失败
- 只读态

根据页面类型补齐：

- 保存失败
- 提交失败
- 批量部分成功
- 下游生成失败
- 数据并发冲突
- 关联对象缺失

权限差异必须在页面中可见：

- 隐藏按钮
- 按钮置灰
- 字段只读
- 字段脱敏
- 页面访问拦截

## 6. 全局工程约定

这一段是一些跨页面都成立的统一口径，避免案例代码和规则文档各说各话。

- 加载中优先用骨架屏
- 编辑页发生并发冲突时，不允许静默覆盖
- 金额默认2位小数，时间默认`YYYY-MM-DD HH:mm:ss`
- Chrome优先，宽度基线按1920评审，1280可完成主路径
- 样例调试控件不得混入业务操作区，应以弱化的悬浮工具形式提供，避免被误读为真实业务按钮
- 样例调试控件在视觉上应显著弱于业务按钮，默认采用低存在感悬浮面板，而不是与主操作同层级的实心按钮
- 样例调试控件不应固定死在右下角遮挡内容；默认应支持鼠标拖动，便于在宽表格、详情页、弹窗演示时自行挪开
- 样例调试控件的拖拽入口默认放在面板头部，不把整个面板都做成可拖拽热区，避免误触

## 7. 这份文档和案例之间怎么配合

可以把两者理解成这样的关系：

- 本文档负责定义“跨项目还成立的稳定规则”
- 案例负责展示“这些规则落到真实页面后大概长什么样”

如果案例里出现了一个好做法，不要直接默认它已经是规则。更稳的流程是：

1. 先看它是不是只服务于某个业务场景
2. 再看它是否在供应商、客户、采购订单等不同案例里都成立
3. 确认它不会限制后续页面扩展
4. 最后再补写回本文档

一句话说：

- **规则先抽象，再落案例**
- **案例先验证，再回写规则**
